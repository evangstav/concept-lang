# concept-lang 0.3.0 — P7: v1 Deletion Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Delete the v1 code that has been living alongside the new v2 code as a safety net since P4, and land the cleanup as `concept-lang 0.3.0`. At the end of this plan, `concept_lang.validator` (v1), `concept_lang.codegen/` (v1), the `_to_v1_concept` adapter in `concept_lang.explorer`, and the `get_dependency_graph` back-compat alias are all gone; `concept_lang.diagrams/` is rewritten to consume the v2 AST directly; `concept_lang.parser`, `concept_lang.models`, `concept_lang.app_parser`, and `concept_lang.app_validator` are kept in place as an explicitly fenced-off legacy subsystem whose sole purpose is to back the v1 app-spec MCP bridge (`tools/app_tools.py`); the v1 test file `tests/test_validator.py` and the `TestV1Adapter` class in `tests/test_explorer.py` are removed; `plugin.json` and `pyproject.toml` are both bumped to `0.3.0`; and `CHANGELOG.md` has a `[0.3.0]` entry plus a fresh `[Unreleased]` placeholder.

**Architecture:** P7 is pure subtraction plus a bounded rewrite. There is no new language surface, no new MCP tool, no new skill, no new validator rule, no new AST type, no new prompt, no new documentation page. The one rewrite is `concept_lang.diagrams/`: `state_machine.py` and `entity_diagram.py` currently import `..models.ConceptAST` (the v1 Pydantic model). P7 rewrites both modules to consume `concept_lang.ast.ConceptAST` (the v2 model) directly so the `_to_v1_concept` adapter becomes dead code and both the adapter and `concept_lang.models` stop being reachable from the v2 path. After the rewrite, `concept_lang.models` is only imported by `concept_lang.app_parser`, `concept_lang.app_validator`, and `concept_lang.tools.app_tools` — the self-contained v1 app-spec bridge — and all four of those files (plus `concept_lang.parser`, which `app_tools` still calls for `.app` file validation) gain a module docstring declaring themselves a fenced-off legacy subsystem that future work must not extend. Every deletion task is a green-to-green transition: the full test suite passes after every commit.

**Tech Stack:** Python 3.10+, Lark, Pydantic 2, pytest, uv. No new runtime dependencies. No new dev dependencies.

**Scope note:** This plan covers **P7 only**. The follow-up work after P7 (app-spec format v2, C8 completeness rule, `OPStep.inputs/outputs` tuple-shape cleanup, the paper's runtime engine layer, marketplace publish) is still deferred and is listed in the "What's next" section at the bottom of this plan.

**Heredoc reminder (P7 execution gotcha).** The `Edit` and `Write` tools have been sandbox-blocked on some worktree paths in prior phases (P4–P6 all hit this). **Design decisions (made and justified up front; later tasks reference them by letter):**

- **(A) Rewrite `concept_lang.diagrams/` to consume v2 instead of deleting it.** The v1 `state_machine.py` and `entity_diagram.py` modules each read a concept's state declarations and actions and emit a Mermaid string. The state they actually need is `decl.name` + `decl.type_expr` for each state declaration, `action.name` + the action's effect clauses, and `concept.name` / `concept.purpose` — all of which exist on the v2 `ConceptAST` with near-identical shapes (v2 `StateDecl` has `name` and `type_expr`; v2 `EffectClause` has `raw`, `field`, `op`, `rhs`). The v1 generators parse the clause strings with regexes against `pre/post.clauses`; the v2 generators can read `action.cases[0].effects` directly. The alternatives were (i) delete `diagrams/` entirely and drop `get_state_machine` / `get_entity_diagram` from the MCP tool surface (rejected: this is a user-visible feature regression with no replacement — the per-concept Mermaid rendering is a core affordance of the `explore` skill and the HTML explorer), (ii) keep `diagrams/` as a v1-only subsystem and let `models.py` + `_to_v1_concept` stay (rejected: leaves v1 dust in the v2 path, keeps the adapter as load-bearing code, and means the grep sweep for "v1 reachable from v2" never goes to zero), (iii) rewrite to v2 (chosen: preserves the user-facing feature, kills the adapter as dead code, and the rewrite is self-contained — roughly 150 lines of Python between the two files). The rewrite is bounded because the two generators do not touch the operational principle, do not touch syncs, and do not touch multi-case actions beyond picking the first case's effects (which is exactly what `_to_v1_concept` already does). P7 treats "pick the first case's effects as the canonical v2 → diagram source" as the rewrite's semantic contract and pins that behavior in the new `tests/test_diagrams.py`.

- **(B) Keep `concept_lang.parser`, `concept_lang.models`, `concept_lang.app_parser`, and `concept_lang.app_validator` as a fenced-off legacy bridge.** These four files back the v1 app-spec functionality exposed by `tools/app_tools.py`. The `.app` format is still v1; a v2 app format is explicitly out of scope for P7 and is slated for a dedicated follow-up plan after 0.3.0 ships. Crucially, `tools/app_tools.py::_load_declared_concepts_v1` calls `parser.parse_file` to read `.concept` files when validating an app spec — which means **the v1 parser is load-bearing for the bridge**, not just the v1 app parser. The inventory grep confirms this: `src/concept_lang/tools/app_tools.py` is the only v2-surface importer of `concept_lang.parser`, and it imports `parse_file` and `ParseError` directly. The alternatives were (i) delete `parser.py` and let `register_app_tools` break (rejected: losing the app-spec tools is a user-visible regression with no replacement, and the `architecture-ide/apps/architecture_ide.app` dogfood file was deliberately preserved by P6 T3 for this reason), (ii) rewrite `app_tools.py` to read v2 `.concept` files via `concept_lang.parse.parse_concept_file` and then adapt to the v1 `ConceptAST` shape that `app_validator.validate_app` expects (rejected: the v1 app-validator expects v1 `SyncClause.trigger_concept` fields which no longer exist in v2; adapting would require either a v2→v1 adapter specifically for the app validator, or porting `app_validator.py` to v2, or dropping `validate_app_spec`'s sync-dependency checking — each of which is scope creep and each of which belongs in the "app format v2" follow-up plan), (iii) move all four files into a `src/concept_lang/legacy/` subdirectory (rejected — see decision (C)), (iv) keep in place and fence off with explicit module docstrings (chosen: zero churn, zero lost history, explicit contract that any future PR touching these files is either an app-spec format v2 migration or a bug). The four files all gain the same header: **"Legacy v1 subsystem — do not extend."** plus a pointer to the app-format-v2 follow-up.

- **(C) Inline docstring fencing, not a `legacy/` subdirectory.** Moving files to `src/concept_lang/legacy/` is cosmetic. The repository already has git history that traces `app_parser.py` through its P4 changes; moving it breaks that trail for no functional gain. Inline fencing keeps blame clean, keeps imports stable (the app bridge still imports `from concept_lang.app_parser import …`), and makes the fencing visible to readers who open the file without needing to navigate a directory tree. The one-line top-of-file module docstring is the canonical fence marker and is backed by the grep sweep in Task 11 — any new `from concept_lang.{parser, models, app_parser, app_validator} import …` import outside of `concept_lang.tools.app_tools` (or, for the inter-legacy imports, the other fenced files themselves) is a fence violation and is caught by the grep.

- **(D) Target version is `0.3.0`, not `0.2.1`.** Deleting `concept_lang.validator` and `concept_lang.codegen` is a hard break for anyone who imported them directly. The `_to_v1_concept` adapter was underscore-private (`_to_v1_concept`, leading underscore) and is technically internal API, but `concept_lang.validator` and `concept_lang.codegen` were public module names exposed in several prior releases' `__init__.py` and had no leading underscore. Semver's patch rule is reserved for backward-compatible bug fixes; removing a public symbol is a breaking change and a minor bump (`0.2.0 → 0.3.0`) is the correct signal. The alternatives were (i) `0.2.1` (rejected: semver violation — removing a public module is not a bug fix), (ii) `0.2.0` (rejected: same semver violation plus would silently overwrite the 0.2.0 release), (iii) `1.0.0` (rejected: the 1.0.0 milestone is reserved for the post-runtime-engine "the plugin is feature-complete" release; 0.3.0 is the cleanup release, not the stability pledge). P7 bumps the version in `.claude-plugin/plugin.json` and `architecture-ide/pyproject.toml` in Task 12 and adds a `[0.3.0]` entry to `CHANGELOG.md` in the same task.

```python
"""
Generate a Mermaid stateDiagram-v2 from a v2 ConceptAST.

Handles two state-declaration patterns:

1. Simple sets:  active: set User
   - Actions that add to `active`  →  [*] --> active
   - Actions that remove from `active`  →  active --> [*]

2. Subset progressions:  named: set T  |  purposeful: named  |  specified: purposeful
   - Builds a chain: [*] → named → purposeful → specified → [*]
   - Each action maps to one primary transition based on what it modifies.
   - Primary transition priority:
     a) base set removed  → base --> [*]
     b) deepest subset added  → parent --> subset
     c) base set added  → [*] --> base
     d) deepest subset removed  → subset --> parent

v2 notes: a v2 action can have multiple cases. This generator looks at
the first case's `effects` list because that is the same contract the
old `_to_v1_concept` adapter exposed (and the contract the existing MCP
`get_state_machine` tool pinned). If an action has no effects or no
cases, it renders as a self-loop on the principal state.
"""

from __future__ import annotations

import re

from concept_lang.ast import Action, ConceptAST


def _first_case_effects(action: Action) -> list[tuple[str, str]]:
    """Return [(field, op), …] for the first case's effect clauses."""
    if not action.cases:
        return []
    case = action.cases[0]
    return [(e.field, e.op) for e in case.effects]


def state_machine(concept: ConceptAST) -> str:
    lines = ["stateDiagram-v2"]

    base_sets: dict[str, str] = {}   # name -> element type
    subsets: dict[str, str] = {}     # name -> parent name
    known_sets: set[str] = set()

    for decl in concept.state:
        m = re.match(r"^set\s+(\w+)$", decl.type_expr)
        if m:
            base_sets[decl.name] = m.group(1)
            known_sets.add(decl.name)
        else:
            parts = decl.type_expr.strip().split()
            candidate_parent = parts[0] if parts else decl.name
            subsets[decl.name] = candidate_parent
            known_sets.add(decl.name)

    if not known_sets:
        # No sets at all — show a flat single-state machine.
        state = f"{concept.name}_active"
        lines.append(f"    state {state}")
        for action in concept.actions:
            lines.append(f"    [*] --> {state} : {action.name}")
        return "\n".join(lines)

    depth: dict[str, int] = {}
    for name in base_sets:
        depth[name] = 1

    changed = True
    while changed:
        changed = False
        for name, parent in subsets.items():
            if parent in depth and name not in depth:
                depth[name] = depth[parent] + 1
                changed = True

    def parent_of(name: str) -> str | None:
        if name in subsets:
            return subsets[name]
        return None

    def deepest(names: set[str]) -> str | None:
        candidates = [(depth.get(n, 0), n) for n in names if n in known_sets]
        return max(candidates, key=lambda x: x[0])[1] if candidates else None

    def shallowest(names: set[str]) -> str | None:
        candidates = [(depth.get(n, 0), n) for n in names if n in known_sets]
        return min(candidates, key=lambda x: x[0])[1] if candidates else None

    for action in concept.actions:
        added: set[str] = set()
        removed: set[str] = set()

        for field, op in _first_case_effects(action):
            if field in known_sets:
                if op == "+=":
                    added.add(field)
                elif op == "-=":
                    removed.add(field)

        if not added and not removed:
            principal = next(iter(base_sets), None) or next(iter(known_sets))
            lines.append(f"    {principal} --> {principal} : {action.name}")
            continue

        if removed & set(base_sets.keys()):
            # Priority a: base set removed → base --> [*]
            base = shallowest(removed & set(base_sets.keys()))
            lines.append(f"    {base} --> [*] : {action.name}")
        elif added - set(base_sets.keys()):
            # Priority b: deepest subset added → parent --> subset
            target = deepest(added - set(base_sets.keys()))
            p = parent_of(target) if target is not None else None
            if p is not None:
                lines.append(f"    {p} --> {target} : {action.name}")
        elif added & set(base_sets.keys()):
            # Priority c: base set added → [*] --> base
            base = deepest(added & set(base_sets.keys()))
            lines.append(f"    [*] --> {base} : {action.name}")
        elif removed - set(base_sets.keys()):
            # Priority d: deepest subset removed → subset --> parent
            target = deepest(removed - set(base_sets.keys()))
            p = parent_of(target) if target is not None else None
            if p is not None:
                lines.append(f"    {target} --> {p} : {action.name}")
        else:
            principal = next(iter(base_sets), None) or next(iter(known_sets))
            lines.append(f"    {principal} --> {principal} : {action.name}")

    return "\n".join(lines)
```

```python
"""
Generate a Mermaid classDiagram from a v2 ConceptAST's state declarations.

Mapping rules:
  name: set T       → entity class T; concept --> T relation
  name: parent      → subset declaration; shown as +name: ⊆ parent (attribute, no arrow)
  X -> set Y        → resolved X class --> Y class (many)
  X -> Y            → resolved X class --> Y class (one)
  anything else     → +name: type attribute on concept class

Single merged class block per concept — no duplicate class entries.
Self-referential relations (concept holds its own element type) are shown
as a containment note rather than a looping arrow.
"""

from __future__ import annotations

import re

from concept_lang.ast import ConceptAST


def entity_diagram(concept: ConceptAST) -> str:
    lines = ["classDiagram"]

    # --- Pass 1: classify all declarations ---

    set_alias: dict[str, str] = {}
    subset_of: dict[str, str] = {}
    known_sets: set[str] = set()

    for decl in concept.state:
        m = re.match(r"^set\s+(\w+)$", decl.type_expr)
        if m:
            set_alias[decl.name] = m.group(1)
            known_sets.add(decl.name)
        elif "->" not in decl.type_expr:
            parts = decl.type_expr.strip().split()
            candidate = parts[0] if parts else ""
            if candidate and candidate[0].islower():
                subset_of[decl.name] = candidate
            known_sets.add(decl.name)
        else:
            known_sets.add(decl.name)

    def resolve_field(name: str) -> str:
        if name in set_alias:
            return set_alias[name]
        if name in subset_of:
            return resolve_field(subset_of[name])
        return name

    # --- Pass 2: build class body attributes and external relations ---

    attrs: list[str] = []
    relation_lines: list[str] = []
    external_classes: set[str] = set()

    for decl in concept.state:
        if re.match(r"^set\s+\w+$", decl.type_expr):
            entity_type = set_alias[decl.name]
            attrs.append(f"        +{decl.name} set~{entity_type}~")
            if entity_type != concept.name:
                external_classes.add(entity_type)
                relation_lines.append(
                    f'    {concept.name} "1" o-- "*" {entity_type} : {decl.name}'
                )
            continue

        if decl.name in subset_of:
            parent = subset_of[decl.name]
            attrs.append(f"        +{decl.name} ⊆ {parent}")
            continue

        m_rel = re.match(r"^(\w+)\s*->\s*(set\s+)?(\w+)$", decl.type_expr)
        if m_rel:
            lhs_type = resolve_field(m_rel.group(1))
            is_many = m_rel.group(2) is not None
            rhs_type = m_rel.group(3)
            card = '"*"' if is_many else '"1"'
            if lhs_type != concept.name:
                external_classes.add(lhs_type)
            if rhs_type != concept.name:
                external_classes.add(rhs_type)
            relation_lines.append(
                f'    {lhs_type} "1" --> {card} {rhs_type} : {decl.name}'
            )
            continue

        attrs.append(f"        +{decl.name} {decl.type_expr}")

    # --- Emit single merged class block ---
    lines.append(f"    class {concept.name} {{")
    lines.append(f'        +purpose "{_truncate(concept.purpose, 45)}"')
    for attr in attrs:
        lines.append(attr)
    lines.append("    }")

    for entity in sorted(external_classes):
        lines.append(f"    class {entity}")

    lines.extend(relation_lines)

    return "\n".join(lines)


def _truncate(s: str, n: int) -> str:
    return s[:n] + "..." if len(s) > n else s
```

## What's next (not in this plan)

After this plan lands and the `p7-v1-deletion-complete` tag is in place, the follow-up work is:

- **App format v2.** The v1 `.app` format is still alive in `0.3.0` via the `register_app_tools` bridge. A dedicated plan migrates it to a v2 AST, writes a v2 validator, rewrites `architecture-ide/apps/architecture_ide.app`, and deletes the entire fenced legacy subsystem (`concept_lang.parser`, `concept_lang.models`, `concept_lang.app_parser`, `concept_lang.app_validator`, and the legacy bits of `concept_lang.tools.app_tools`).

- **App-spec pytest coverage.** The P6 baseline had no `TestAppTools` class and P7 did not add one. A follow-up plan (bundled with "App format v2" above or shipped separately) adds real unit tests for `list_apps`, `read_app`, `write_app`, `validate_app_spec`, and `get_app_dependency_graph`.

- **C8 completeness rule.** The unreferenced-state warning rule was deferred from P2 because it is hard to express as a minimal positive fixture. A follow-up plan reintroduces it once the authoring pipeline has enough signal to distinguish "unused state" from "state referenced by a sync-level query".

- **`OPStep.inputs/outputs` tuple-shape cleanup.** The AST still stores operational-principle input / output fields as `list[tuple[str, str]]` rather than `list[TypedName]`. The cleanup is low-risk but not urgent and has been carried across P3, P4, P5, P6, and now P7 without being done.

- **Paper-aligned code generator.** `concept_lang.codegen/` was deleted in Task 6 because it was dead code from the v1 era. A paper-aligned code generator (that reads v2 AST and emits per-concept implementation stubs in a target language) is a plausible future workstream but is not in any committed roadmap.

- **Paper-aligned runtime engine (Layer 2).** The paper's §6 and Appendix A describe a full synchronization runtime with action graphs and flow-ID tracking. This is explicitly out of scope for the language + tooling spec and is slated for its own follow-up spec.

- **`get_workspace_graph` clean description.** After P7, the `workspace_tools.py` description string no longer mentions "Replaces the old `get_dependency_graph`." A future docs pass may rewrite the description again to talk about syncs-as-edges without the historical note.

- **Marketplace publish.** The user handles release distribution manually; no plan needed.
